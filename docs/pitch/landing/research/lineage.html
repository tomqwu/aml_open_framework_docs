Walk-back · 2026 edition

Trace every alert. Down to the row.

When an examiner asks "show me why alert N fired", most AML platforms can answer at the level of "this customer, this rule, this severity." The framework answers at the level of this file path, this column, this row index, this rendered SQL, this rule version, this case_id, this STR — every link hash-stamped, every chain reproducible. This doc walks the chain link by link.

Audience · MLRO · CCO · 2LoD · Internal Audit · Examiner Anchors · BCBS 239 P3-P5 · FinCEN April 2026 NPRM · OSFI E-23 · SR 26-2 Last refresh · May 2026

The question regulators actually ask.

Walk into any AML examination from the last five years and the question lands in the first hour: "Show me alert N. Show me the rule that fired it. Show me the data the rule looked at. Show me the version of the rule on that day. Show me the analyst's decision and when they made it. Show me that you can produce the same result if I ask you to re-run." Every word of that sentence is a different stamp on a different artifact, and most platforms keep them in different systems — separately searchable, never cross-walkable. The framework treats the answer as a single chain.

The chain has 7 links Source file → Contract → DuckDB table → Rule (SQL or callable) → Alert (with matched rowids) → Case → Investigation / STR. Each link carries its own hash. Each link is reproducible from the spec + data + as_of triple. Same inputs, same hashes — every time.

What gets stamped.

Every alert decision the engine writes carries 12 fields. The shape lives in src/aml_framework/engine/audit.py; the canonical reader is walk_lineage(run_dir, case_id). Twelve fields, one chain:

Stamped · per decision

The 12-field per-decision payload.

case_id
Deterministic from rule_id + customer_id + window_end. Same inputs → same case_id.
rule_id
The human-authored rule name from the spec. Stable across versions.
rule_version
SHA-256[:16] over the rule's serialised content. A threshold change, filter change, or even field reordering produces a different version. Stamped on every decision event (PR-LIN-3) — not just case_opened — so an escalate or close event proves the rule version was unchanged mid-investigation.
spec_content_hash
SHA-256 of the entire spec snapshot at execution time. The Manifest hash MLRO sign-off chains to via aml attest.
engine_version
Version of the runtime that produced the run. Pinned in manifest.json.
as_of
The wall-clock anchor the spec was evaluated against. Decision timestamps default to as_of for run-to-run determinism.
source_path
Per-contract logical source — file path, warehouse table, or "synthetic". Added in PR-LIN-2 so the auditor can answer "which file did this row come from?"
schema_columns + schema_hash
The contract's column list at load time, plus a 16-hex SHA-256 over the sorted column names. A column rename or drop produces a different schema_hash even when row content is unchanged. Answers "why did the same source produce different alerts after a Tuesday DDL change?"
row_count + content_hash
Per-contract row count and SHA-256 over the canonicalised, sorted row content. Two runs with the same source but different content produce different content_hashes — the data-drift signal.
matched_row_ids
DuckDB rowids of the source rows that fired the rule. Added in PR-LIN-4 across aggregation_window / custom_sql / list_match / network_pattern. Lets the dashboard show "these are the 7 transactions that triggered this alert" rather than just "this customer."
rule_sql
The post-substitution rendered SQL the engine actually executed. Captured by record_rule_sql() on every rule, surfaced into the lineage chain by PR-LIN-1's walk_lineage() extension.
schema_version
Constant 2. Lets future readers detect old runs and apply default values for fields that didn't exist yet.

The walk-back.

A 2LoD reviewer pastes a case_id into the Audit & Evidence page (or, deeper, the new Lineage Explorer page). The framework returns the chain.

walk_lineage(run_dir, "C0042-cash_structuring-001")

A worked example.

Case
{ case_id: "C0042-...", rule_id: "cash_structuring", rule_version: "a1b2c3d4e5f6g7h8", queue: "L1" }
Run anchors
{ run_dir: "run-20260507T101500Z", spec_content_hash: "9f3a...", engine_version: "1.0.0", as_of: "2026-05-07T00:00:00Z" }
Source files (1 of N)
{ contract_id: "txn", source_path: "data/input/txn.csv", row_count: 12847, schema_hash: "44a8...", content_hash: "b2c1...", schema_columns: [txn_id, customer_id, amount, ...] }
Rule SQL
SELECT customer_id, SUM(amount) AS sum_amount, COUNT(*) AS count, '2026-04-07' AS window_start, '2026-05-07' AS window_end FROM txn WHERE channel='cash' AND amount >= 9000 AND booked_at >= TIMESTAMP '2026-04-07 00:00:00' AND booked_at < TIMESTAMP '2026-05-07 00:00:00' GROUP BY customer_id HAVING SUM(amount) > 25000
Matched source rows
[3742, 5891, 7123, 9445, 11203, 11856, 12410] — seven rowids from txn that contributed to the aggregation. The dashboard renders them in an AG Grid sliced from the in-memory frame.
Decision timeline
case_opened (rule_version a1b2c3d4...) → escalate (same rule_version, queue L2) → escalated_to_str (same rule_version, queue STR_FILING). PR-LIN-3 stamps rule_version on every event so the chain is unbroken across the investigation.
Export
One click downloads lineage_C0042-...json — round-trippable, attachable to a SAR, hash-comparable against an earlier export of the same case_id from the same run.

Reproducibility, proven.

The chain only matters if it's stable. The framework's reproducibility contract — same spec + same data + same seed → identical hashes — is enforced as a test (test_run_is_reproducible in tests/test_audit_decision_metadata.py). Re-run the same spec against the same data with the same as_of and seed, and every artifact in the run dir is byte-identical: the decisions.jsonl hash, the alerts/<rule_id>.hash, the manifest.json's spec_content_hash and decisions_hash, the rendered SQL files, the input manifest's content hashes. If anything in the input changed, exactly one hash diverges, and the chain says where.

Why MLROs care The hash chain is the primitive. aml attest chains a signed MLRO sign-off to spec_content_hash. aml run --strict refuses to execute against unattested specs. Re-running with --seed 42 after an examiner asks "show me again" produces the same output — verifiable bit-for-bit.

What this replaces.

Vendor "audit log" features typically capture the fact of a decision (analyst X clicked escalate at time T) but not the chain underneath the decision. The rule version isn't stamped. The source path isn't recorded. The matched rows are gone the moment the alert lands. The rendered SQL was never written down. Re-running to verify is impossible because the data has rotated and the rule has been edited.

The framework keeps every link addressable from case_id alone. The 12 fields above are the difference between "we have a log" and "we have a chain." A log answers did it happen?; a chain answers why did this specific output exist on this specific day given this specific input?

Where to see it live.

Three surfaces, each at a different depth:

  1. Case Investigation → "Why this fired" panel. The in-workflow answer for the L1/L2 analyst. Shows matched-row count + severity + rule_version + the rule SQL in a collapsible block, above the existing transaction timeline. Picks up the most-asked question without leaving the case.
  2. Audit & Evidence → Lineage walk-back panel. The auditor's read. Paste a case_id, get the full chain — case anchors, run anchors, source provenance (path + schema hash + content hash + columns), rule SQL, matched source rows from the in-memory txn frame, and the full decision timeline.
  3. Lineage Explorer (page #32, new). The deep drill. Mermaid graph of the 7-link chain, the same data the walk-back panel surfaces, and a JSON download for offline review. Deep-linkable from any case_id via ?case_id=....

Cited regulator anchors.

  1. BCBS 239 Principle 3 — Accuracy and Integrity. "Risk data should be aggregated on a largely automated basis to minimise the probability of errors." A walk-back chain is what aggregation discipline looks like in evidence form. — BCBS 239, BIS, January 2013.
  2. BCBS 239 Principle 4 — Completeness. "Banks should be able to capture and aggregate all material risk data across the banking group." Source-path + schema_hash stamps make the gap detectable, not silent. — BCBS d559 progress report, 2023.
  3. FinCEN April 2026 NPRM — Effectiveness. Measures effectiveness per investigation, requires walk-back to the underlying alerts. The framework's aml effectiveness-pack generator pulls the lineage chain into the regulator-format JSON. — FinCEN.
  4. SR 26-2 / OCC Bulletin 2026-13 — Model Risk Management. Requires a documented audit trail from model output to model input to model version. The lineage chain is the audit trail. The MRM bundle generator consumes walk_lineage() output. — Federal Reserve SR 26-2.
  5. OSFI E-23 (revised) — Model Risk Management. Requires institutions to demonstrate "consistent, reproducible outcomes" from models — the framework's reproducibility contract proves this property as a CI-enforced test. — OSFI E-23.
