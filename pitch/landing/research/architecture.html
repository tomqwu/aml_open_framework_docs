Architecture · design rationale

One Manifest. Four layers. No drift.

Policy, data contracts, detection rules, case workflow, and regulator mapping live in one versioned document — the Compliance Manifest (an aml.yaml file, for engineers). Every runtime artifact — SQL, DAGs, data-quality tests, dashboards, alert payloads, audit ledger entries — is generated from the Manifest, never hand-written in parallel. This is what kills the drift that causes AML enforcement actions.

Pattern · spec-as-source-of-truth (cf. Terraform / dbt / Kubernetes) License · Apache 2.0 / MIT-compatible Runtime · pip-installable · DuckDB + FastAPI + Streamlit

Determinism & reproducibility.

Every rule execution records four hashes:

  1. manifest_version — git SHA of the Compliance Manifest (aml.yaml) plus a content hash. What policy ran?
  2. input_hash — hash of the ordered input rows (or a snapshot ID). What data did it see?
  3. output_hash — hash of the alert set. What did it produce?
  4. engine_version — version of this framework. What code executed?

An auditor can re-run any historical execution and verify the output hash matches. If it doesn't, the chain of custody is broken and the run is flagged. This is what FCA's Dear-CEO Letter Mar 2024 asked for and almost no commercial platform delivers as a published guarantee.

Why not a rules engine in application code?

A pure Python or Java rules engine lets any developer tweak detection logic in a Monday-morning hotfix, and the CCO only hears about it three audits later. Declarative specs with PR-based review force a control point: policy changes go through compliance sign-off before they change pipeline behaviour. The same argument is why Terraform, dbt, and Kubernetes manifests won — spec > code for regulated change.

Extensibility: two escape hatches.

Spec-as-source-of-truth is strict. Two intentional escape hatches preserve the audit trail:

  • custom_sql on a rule — engineer drops in handwritten SQL when declarative primitives aren't expressive enough. The spec still carries the regulation citation, severity, workflow, and evidence list.
  • python_ref on a rule — points at a Python callable (e.g. an ML scorer) that returns an alert set. The spec captures the model id and version so model risk management can validate it.

Both escape hatches mean some generation properties weaken (e.g. the control matrix can't auto-extract thresholds from custom SQL), but the policy layer is preserved and the audit trail still works.

What this framework is not.

  • Not a replacement for a core banking system or transaction store. It reads from whatever warehouse you already have.
  • Not a certified detection-model catalogue. Rules in the example specs are illustrative starting points, not validated typologies — every Tier-1 deployment will tune them for its own risk model.
  • Not a SAR filing service. It produces a well-formed regulator bundle; the actual submission step is institution-specific.
