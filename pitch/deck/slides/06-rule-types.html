Discriminated union · type-driven dispatch

Four ways to express a detection.

aggregation_window
CASH STRUCTURING

Threshold-based aggregations over a sliding window.

# Sum of cash deposits per customer
# over 7 days, alert if > $9,000.
type: aggregation_window
window: 7d
filters:
  channel: cash
aggregate:
  field: amount
  op: sum
having: "> 9000"
Use when a regulator gave you a number and a window — the cleanest, most spec-readable form.
custom_sql
RAPID PASS-THROUGH

Bespoke DuckDB SQL with placeholders.

# Cash-in > $50k followed by
# e-transfer-out within 48h.
type: custom_sql
sql: |
  SELECT customer_id, SUM(amount) ...
  WHERE booked_at > '{recent_start}'
    AND channel IN ('cash','eft')
  GROUP BY customer_id
  HAVING SUM(amount) > 100000
Use when the pattern needs joins or self-references the spec language can't express.
python_ref
ML RISK SCORER

Callable Python ref — ML, network analysis, anything.

# Resolves to a callable in
# aml_framework.models.scoring
type: python_ref
callable: aml_framework.models
  .scoring.ml_risk_score
params:
  threshold: 0.85
  model_version: "v3.2.1"
Use when ML or graph-walk logic is needed; spec still references a versioned, hash-stamped callable.
list_match
SANCTIONS / PEP / ADVERSE MEDIA

Reference-list screening with exact or fuzzy match.

# Match customer.full_name against
# OFAC SDN — fuzzy, threshold 0.8.
type: list_match
list: OFAC_SDN
field: full_name
match: fuzzy
threshold: 0.8
severity: critical
Use when screening against published lists (sanctions, PEP, adverse media, sanctioned wallets).
All four execute in the reference engine. One discriminator field, one dispatch table, no new code per rule.
